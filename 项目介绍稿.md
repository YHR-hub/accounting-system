- 写了 96 个 pytest 单元测试，覆盖所有核心业务逻辑，10 秒全通过

## 五、面试口述稿（3-5 分钟完整版）

> **面试官追问细节。**

### 5.1 为什么做这个项目？

"其实最开始是为了解决一个实际问题——我身边有做会计的朋友，他们用的金蝶用友要么太贵要么太复杂。所以我想要做一个轻量级的、开箱即用的替代品。

后来做着做着发现，这个项目其实能很好地展示软件工程的全栈能力：数据库设计、业务建模、前端交互、测试、打包部署，全都有。"

### 5.2 技术难点

**（1）借贷记账法的核心逻辑**

"会计的核心是『有借必有贷，借贷必相等』。我在数据库层面用 `journal_entries` 表来存每一笔分录，`debit` 和 `credit` 两个字段。创建凭证时做事务校验：所有 debit 之和必须等于所有 credit 之和，不相等就回滚。这保证了账目的完整性。"

**（2）FIFO 存货成本核算**

"进销存模块里，我实现了先进先出法。出库时，按入库时间顺序逐笔扣减库存，先入库的先出库，成本也是按先入库的价格计算。这个逻辑涉及时间序的遍历和数量拆分，相对复杂。"

**（3）区块链账本**

"我实现了基于 SHA-256 的区块链结构。每个区块包含索引、时间戳、前一区块哈希和交易数据。添加新块时做 PoW 工作量证明（前导零挖矿）。整条链可以导出/导入 JSON 并做完整性校验，确保历史财务数据不可篡改。"

**（4）年度化个税计算**

"中国的个税是超额累进税率，每月预扣预缴需要年度化计算：先把月应税收入×12 得到年应税收入，按 7 级累进税率算年税，再÷12 得到月税。这个算法既要符合税法规定，又要处理起征点（5000/月）和专项扣除。"

**（5）代码重构**

"这是我能体现工程能力的地方。原来是一个 4858 行的后端单文件加一个 3700 行的 GUI 单文件。我分两步做了重构：

第一步，把后端按业务域拆成 27 个模块——每个模块只做一件事，比如 `vouchers.py` 只管凭证、`payroll.py` 只管薪资、`blockchain.py` 只管区块链。每个模块都有清晰的边界和导入关系。

第二步，把 GUI 从一个大文件拆成 24 个 Mixin 组件，一个标签页一个文件。主应用类通过多重继承把所有 Mixin 拼起来，`app.py` 从 3700 行缩到 300 行。这其实就是 Django MTV 的那种关注点分离思路，只不过应用在了桌面 GUI 上。

重构过程中所有交叉引用要正确、96 个测试不能引入回归 bug。最终 58 个 Python 文件、约 10000 行代码，全都跑通。"

## 八、常见问题 QA

**Q: 这个项目用了什么框架？**
A: 没有用任何商业框架。GUI 用 Python 自带的 tkinter，ORM 用原生 sqlite3 模块，图表用 matplotlib，AI 调用用 requests 库。

**Q: 数据库怎么设计的？**
A: 20 张表，核心是 accounts（科目表）、vouchers（凭证头）、journal_entries（分录明细）。其他业务表如 fixed_assets、inventory_transactions、payroll_records、bank_statements 等围绕核心展开。

**Q: 和其他会计软件比有什么优势？**
A: 轻量（43MB）、免费、开源、可定制。功能覆盖金蝶用友的基础模块，额外提供 AI 分析、区块链验真、ESG 报告等特色功能。

**Q: 有没有线上版本？**
A: 目前是桌面版本。架构上后端逻辑和 GUI 已经完全分离，未来可以基于 accsys 包开发 Web 版本（Flask/Django）。
