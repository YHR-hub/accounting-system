【开场 — 30秒】
----------------------------------------------------------------

大家好，我今天要介绍的项目是"会计系统专业版"——
一个我从零开发的、全功能的中文会计桌面软件。

它不是一个 demo，而是一个真正能用的产品——
从数据库设计、业务逻辑，到前端交互、测试、打包，完整走通了。

先给大家看一组数字：
29 个功能模块，27 个后端模块，24 个 GUI 标签页，
58 个 Python 文件，96 个单元测试，
4800 条凭证的高压下全部通过。

下面我从三个角度展开：做什么、怎么做、做得怎么样。


【第一部分：做什么 — 功能全景 — 1 分钟】
----------------------------------------------------------------

这个系统覆盖了一个会计软件需要的全流程，我分四层来介绍：

基础功能：
  凭证录入（借贷记账法）、科目管理、总账、试算平衡

三大报表：
  资产负债表、利润表、现金流量表
  其中现金流表实现了间接法和直接法两种模式

进阶模块：
  固定资产管理，支持三种折旧方法（年限平均、双倍余额递减、年数总和）
  增值税和个税计算（7 级超额累进税率，年度化算法）
  多币种核算、期末自动结账

特色功能：
  这是我觉得最有意思的部分——
  AI 云助手（接入大模型，用自然语言查账、分析报表）
  区块链账本（SHA-256 链式结构，PoW 工作量证明，链上数据不可篡改）
  ESG 报告（环境、社会、治理三类 15 项指标，自动评分）
  进销存管理（FIFO 先进先出成本核算）
  薪资管理（社保、公积金、个税自动计算，一键生成工资凭证）
  银企对账（导入银行流水，自动匹配凭证）
  财务预警（自定义阈值，自动检查关键财务指标）
  项目会计（按项目独立核算损益）
  多用户权限（管理员、会计员、查询员三种角色）
  暗色模式（完整主题切换）

总共 29 个模块，所有逻辑都放在一个统一的 accsys 包里。


【第二部分：怎么做 — 架构设计 — 2 分钟】
----------------------------------------------------------------

（屏幕展示项目结构树）

技术选型上，我刻意没有用任何商业框架：
  后端：纯 Python，没用 Django，没用 ORM
  数据库：SQLite，20 张表，原生 sqlite3 模块
  GUI：Python 自带的 tkinter
  可视化：matplotlib
  AI 调用：requests 库
  打包：PyInstaller 单文件 EXE

这个选择的目的，是证明我从底层到应用都有掌控力。

数据库设计上：
  核心是三张表：accounts（科目表）、vouchers（凭证头）、
  journal_entries（分录明细）
  然后围绕核心扩展出 17 张业务表——
  固定资产、库存交易、银行流水、薪资记录、项目、审计日志等等

架构演进——这可能是我最自豪的部分：

  最初是单体开发，后端一个文件 4858 行，GUI 一个文件 3700 行。
  能跑，但绝不是好的工程实践。

  所以我分两步做了重构：

  第一步，把后端按业务域拆成 27 个模块。
  每个模块只做一件事：vouchers.py 只管理凭证，
  payroll.py 只管理薪资，blockchain.py 只管理区块链。

  第二步，把 GUI 从一个大文件拆成 24 个 Mixin 组件。
  一个标签页一个文件，主应用类通过多重继承把它们拼起来。
  app.py 从 3700 行缩到了 300 行。

  这其实就是 Django 的 MTV 或 MVC 那种关注点分离的思路，
  只是应用在了桌面 GUI 上。

  拆完之后，我又做了一个关键的验证：
  基于 accsys 包，用 Flask 搭了一个 Web 版。
  只花了 150 行代码就搭出了完整的 Web 界面——
  仪表盘、凭证列表、科目余额、三张报表、JSON API。
  这证明了后端逻辑和前端是完全解耦的。

  同一套 accsys 包，上面可以接 tkinter 桌面端，
  也可以接 Flask Web 端，未来还可以接 REST API 移动端。


【第三部分：做得怎么样 — 质量与测试 — 1.5 分钟】
----------------------------------------------------------------

一个项目，功能再多，如果质量不过关，就是空中楼阁。
所以我在测试上花了相当的精力。

单元测试：
  96 个 pytest 测试用例，覆盖了所有核心业务逻辑——
  数据库初始化、凭证创建、报表生成、税率计算、
  折旧计算、库存出入库、薪资计算、用户认证等等
  10 秒全部跑完。

高压测试——这是我最想展示的：
  我写了一个专门的压测脚本，同时灌入 4800 条凭证、
  约 10000 条分录，然后执行：
    科目余额计算 ×10  —— 0.06 秒
    三张报表生成 ×5   —— 0.18 秒
    财务比率分析 ×20  —— 0.13 秒
    500 次数据库连接   —— 0.62 秒
    库存 200 笔操作    —— 1.37 秒

  所有操作都在毫秒级完成，没有崩溃、没有卡顿。
  数据库文件只膨胀到 1.0 MB。

  这说明什么？说明架构的健壮性——
  不是只有几条数据时能跑，上了量依然稳得住。

CI/CD：
  配置了 GitHub Actions，每次推送自动跑 pytest
  支持 Python 3.10 / 3.11 / 3.12 三个版本


【第四部分：技术亮点 — 1 分钟】
----------------------------------------------------------------

我快速讲三个技术实现上的亮点：

1. 借贷记账法
   会计的核心是"有借必有贷，借贷必相等"。
   创建凭证时，我做了事务级的校验——
   所有分录的借方合计必须等于贷方合计，
   不相等就回滚整个事务。

2. FIFO 成本核算
   进销存里，我实现了先进先出法。
   出库时按入库时间顺序逐笔扣减，
   先入库的先发出，成本按先入库的价格确定。
   这个逻辑涉及时间序列的遍历和数量拆分。

3. 年度化个税
   中国个税是 7 级超额累进税率。
   月预扣 =（年应税收入 × 年税率 - 速算扣除数）÷ 12
   需要处理起征点（5000 元/月）和社保公积金扣除。

【如果被追问】
  - 为什么不用 Django/Flask 做后端？
    → 这是桌面应用，不需要 HTTP 服务。但我也证明了
       accsys 包可以直接接 Flask，需要时随时可以 Web 化。

  - 区块链在这个系统里的实际意义？
    → 财务数据防篡改。每一笔凭证上链后，
      任何改动都会破坏哈希链，能立刻被验证发现。
      这是给审计合规增加一层技术保障。

  - 最大的挑战是什么？
    → 不是某个具体算法，而是重构的过程。
      把一个能跑的系统拆开，同时保证所有交叉引用正确、
      96 个测试不引入回归 bug，这需要非常小心。
